# 제 3 강 - UX 리서치 

## 사전 질문
1. 서비스를 사용하면서 "이건 사용자를 전혀 이해하지 못하고 만들었구나"라고 느낀 경험이 있나요? 있다면 그 이유는 무엇이었나요?
    - 애플 사진첩
        * icloud에 저장되서 모든 디바이스에 동기화 되는건 좋음
        * MacOS에서 파인더로 이동 정리등이 쉽지않음
        *  아이폰 상태에서 정리는 쉬움 - 맥북사용자를 위한 인터페이스가 아님.
2. "사용자 입장에서 생각해봐"라는 말을 들었을 때, 정말로 사용자 입장이 될 수 있다고 생각하나요?
    - 서비스를 직접 만든 입장이 아니라면, 충분히 가능할 것 같습니다.
    - 그냥 내가 사용하는 방식 자체가 사용자 일테니까요.

## UX 리서치

### 정의
UX 리서치는 사용자의 니즈, 행동, 페인 포인트를 이해하기 위해 체계적으로 데이터를 수집하고 분석하는 프로세스입니다. 제품이나 서비스를 만들 때 **추측이 아닌 사용자 데이터에 기반한 의사결정**을 하기 위한 필수 과정입니다.

### UX 리서치의 중요성

| 항목 | 설명 |
|------|------|
| **사용자 중심 설계** | 실제 사용자의 관점에서 문제를 정의하고 해결책을 찾음 |
| **비용 절감** | 개발 후 문제 발견보다 사전에 문제를 파악하여 재작업 감소 |
| **만족도 향상** | 사용자 니즈에 맞춘 설계로 제품 만족도 증가 |
| **시장 경쟁력** | 사용자 인사이트 기반의 차별화된 기능 제공 |

### UX 리서치의 종류

#### 1. **정량적 리서치 (Quantitative Research)**
사용자 행동을 **숫자와 통계로 측정**하는 방식

**특징**:
- 큰 표본 크기로 통계적 유의성 확보
- 객관적이고 재현 가능한 데이터
- "얼마나 많은 사용자가" 문제를 겪는지 파악

**주요 방법**:
- **설문조사 (Survey)**: 다양한 사용자로부터 의견 수집
- **분석 (Analytics)**: 웹/앱 사용 데이터 분석 (클릭, 체류시간 등)
- **A/B 테스트**: 두 가지 안을 비교하여 효과 측정

#### 2. **정성적 리서치 (Qualitative Research)**
사용자의 **동기, 감정, 이유를 깊이 있게 탐색**하는 방식

**특징**:
- 소규모 표본이지만 상세한 인사이트
- "왜(why)" 질문에 답함
- 사용자의 목소리를 직접 들음

**주요 방법**:
- **인터뷰 (User Interview)**: 일대일 심화 인터뷰
- **포커스 그룹 (Focus Group)**: 소규룹 사용자와의 집단 토의
- **사용성 테스트 (Usability Test)**: 실제 사용 장면 관찰
- **관찰 (Observation)**: 자연스러운 환경에서 사용자 행동 관찰

### 리서치 프로세스

#### Step 1: **리서치 계획**
```
질문 정의 → 목표 설정 → 대상자 선정 → 방법론 결정 → 일정 수립
```

**정하기**:
- "무엇을 알고 싶은가?"
- "누구에게 물어볼 것인가?"
- "어떤 방법으로 할 것인가?"

#### Step 2: **데이터 수집**
- 정성: 인터뷰, 관찰, 포커스 그룹
- 정량: 설문, 분석, 테스트

#### Step 3: **데이터 분석**
- **정성**: 코딩 (Coding) → 패턴 도출 → 테마 추출
- **정량**: 통계 분석 → 결과 해석

#### Step 4: **인사이트 도출**
주요 발견 사항을 정리:
- "사용자는 X 때문에 Y를 어려워한다"
- "Z 기능이 가장 중요한 필요성이다"

#### Step 5: **실행 및 반복**
- 인사이트를 디자인/개발에 반영
- 변화 측정 및 다시 리서치

### 주요 리서치 방법 상세

#### 사용자 인터뷰 (User Interview)

**목적**: 사용자의 생각, 경험, 니즈를 깊이 있게 이해

**방식**:
- **구조화된 인터뷰**: 미리 정한 질문지 사용
- **반구조화된 인터뷰**: 기본 질문은 있으되 유연하게 진행
- **비구조화된 인터뷰**: 자유로운 대화

**팁**:
- 열린 질문 사용: "어떻게 생각하세요?" (O) vs "이거 좋나요?" (X)
- 사용자의 말을 끝까지 들음
- "왜?"를 3~5번 반복 (5 Why's)

#### 사용성 테스트 (Usability Test)

**목적**: 실제 사용 장면에서의 문제점 파악

**진행 방식**:
1. 사용자에게 과제 부여 ("예약을 해보세요")
2. 사용자가 작업하는 모습을 관찰
3. 어려움, 혼란, 오류 기록
4. 작업 후 인터뷰

**측정 지표**:
- 완료율: 과제를 완료했는가?
- 완료 시간: 얼마나 빨리 완료했는가?
- 에러: 실수가 몇 개 나왔는가?
- 만족도: 사용자 만족도는?

#### 설문조사 (Survey)

**목적**: 많은 사용자로부터 정량적 데이터 수집

**효과적인 설문 작성**:
- 간결한 문장 사용
- 한 질문에 한 가지만 물음
- 편향되지 않은 선택지 제공

#### 분석 (Analytics)

**추적 항목**:
- **도달률**: 페이지 방문자 수
- **체류시간**: 평균 체류 시간
- **클릭 경로**: 사용자의 이동 경로
- **이탈률**: 이탈 지점 파악
- **전환율**: 목표 달성 비율

### UX 리서치 시 주의점

> [!warning]
> **확인 편향 (Confirmation Bias) 피하기**
> - 자신의 가정을 확인하려는 질문만 하지 말 것
> - 반대 의견도 수용하는 자세 필요

> [!important]
> **최소 표본 크기**
> - 정성: 5~8명이면 충분 (그 이상은 반복)
> - 정량: 최소 100명 이상 권장

> [!note]
> **사용자 모집의 어려움**
> - 내부 직원이나 가족과의 테스트는 피할 것
> - 실제 타겟 사용자와 테스트해야 함

### 리서치 결과 활용

#### 페르소나 (Persona) 작성
특정 사용자 그룹을 대표하는 가상의 인물:
- 이름, 나이, 직업
- 사용 목적과 목표
- 페인 포인트와 기대사항

#### 사용자 저니 맵 (User Journey Map)
사용자가 서비스를 이용하는 전체 과정을 시각화:
- **Touchpoint**: 사용자가 만나는 모든 접점
- **감정**: 각 단계에서의 감정 변화
- **Pain Point**: 문제가 되는 부분
- **Opportunity**: 개선 가능한 기회

## 과제

### 연구 과제
1. 자주 쓰는 앱의 첫 화면을 캡처해서 AI에게 보여주고 5초 테스트를 해보십시오. "이게 뭘 하는 서비스인가요?", "가장 먼저 눈에 띄는 건 뭔가요?" 답변이 내 예상과 다르다면, 그 차이는 어디서 오는 걸까요?
2. 같은 앱의 문제점을 파악하기 위한 설문 질문 3개를 만들고 AI에게 검토를 요청해보십시오. 강의의 "좋은 질문 조건"을 지켰다고 생각했는데 잘못된 부분이 있다면, 왜 못 찾아냈을지 생각해 봅니다.
3. 그 앱과 경쟁 앱 하나를 골라 같은 기능의 화면을 비교해보십시오. "왜 다르게 만들었을까?"를 설명하려 할 때, 막히는 부분은 어디인가요?


### 실전 과제
1. 서비스 하나를 선택하고, 이 서비스가 사용자에게 제공하려는 핵심 가치를 분석하십시오. 그 핵심 가치를 저해하는 사용자 경험 중 하나를 선택하여 문제를 구체적으로 정의하십시오.
2. 이 문제가 What/Why의 문제인지 How의 문제인지 판단하고, 이것을 해결하기 위해 필요한 UX리서치 방법 중 적합한 방법을 선택하십시오. 다른 방법이 아닌 이 방법을 선택한 이유를 구체적으로 작성하십시오.
3. 선택한 방법으로 리서치를 수행하기 위한 계획을 작성하십시오. 작성한 계획을 스스로 검토하여 보완이 필요한 부분을 개선한 뒤 최종 계획을 정리하십시오.
