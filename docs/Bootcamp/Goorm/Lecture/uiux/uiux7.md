# 제 7 강 - UX 라이팅

## 사전 질문
1. 서비스를 사용하며 이해가 가지 않는 문구를 만나게 되면 어떤 경험을 하게 되나요? 이런 경험이 서비스 이용자에게 어떤 영향을 미치게 될까요?
    - 어떤 내용인지 찾아보게 되고, 끝내 이해할 수 없다면 이탈하게 될 것 같습니다.

2. 누르기 안심이 되는 문구는 어떤 특징이 있나요? 반대로 누르지 않고 싶은 문구들은 어떤 특징이 있나요? 이 둘을 비교해 설명해 봅니다.
    - 직관적이고 어떤 일이 일어날지 예측가능한 메타포들은 쉽게 안심이 되는것 같고
    - 예측할 수 없고 어떤일이 일어날지 모르는 메타포들은 누르기 꺼려 지는것 같습니다.

## UX 라이팅

**UX 라이팅(UX Writing)**은 사용자가 제품을 쉽고 효과적으로 사용할 수 있도록 돕는 텍스트를 작성하는 일입니다. 버튼 레이블, 오류 메시지, 온보딩 문구 등 사용자 인터페이스에 나타나는 모든 텍스트가 UX 라이팅의 영역입니다.

**UX 라이팅의 정의:**
```
사용자 인터페이스에서 사용자와 제품 간의 대화를 설계하고 작성하는 작업.
명확하고, 간결하며, 유용한 텍스트로 사용자 경험을 향상시킨다.
```

### UX 라이팅이 중요한 이유

**1. 사용자 이해를 돕는다**
- 기능과 목적을 명확히 전달
- 다음 행동을 안내
- 불확실성 제거

**2. 신뢰를 구축한다**
- 정직하고 투명한 커뮤니케이션
- 예측 가능한 결과 제시
- 일관된 톤 유지

**3. 행동을 유도한다**
- 명확한 CTA(Call To Action)
- 동기부여하는 메시지
- 마찰 지점 제거

**4. 브랜드를 표현한다**
- 브랜드 개성 전달
- 감정적 연결
- 차별화된 경험

### UX 라이팅의 핵심 원칙

**1. 명확성 (Clarity)**
- 사용자가 즉시 이해할 수 있는 언어
- 전문 용어나 애매한 표현 지양
- 구체적이고 직접적인 표현

```
❌ "프로세스를 진행하시겠습니까?"
✅ "회원가입을 완료하시겠습니까?"

❌ "작업이 성공적으로 수행되었습니다."
✅ "파일이 저장되었습니다."
```

**2. 간결성 (Conciseness)**
- 필요한 만큼만 작성
- 불필요한 단어 제거
- 핵심 메시지에 집중

```
❌ "만약 계속 진행하고 싶으시다면 아래 버튼을 클릭해주세요."
✅ "계속하기"

❌ "귀하의 이메일 주소를 입력해주시기 바랍니다."
✅ "이메일 주소"
```

**3. 유용성 (Usefulness)**
- 사용자가 필요로 하는 정보 제공
- 다음 행동 안내
- 문제 해결 방법 제시

```
❌ "오류가 발생했습니다."
✅ "이메일 형식이 올바르지 않습니다. '@'를 포함해주세요."

❌ "성공"
✅ "비밀번호가 변경되었습니다. 새 비밀번호로 로그인해주세요."
```

**4. 인간적 (Conversational)**
- 자연스러운 대화체
- 친근하고 공감하는 톤
- 과도한 격식 지양

```
❌ "귀하의 요청이 접수되었사오니 잠시 기다려주시기 바랍니다."
✅ "요청을 처리하고 있어요. 잠시만 기다려주세요."

❌ "입력값이 유효하지 않습니다."
✅ "다시 확인해주세요."
```

## UX 라이팅의 7가지 적용 영역

### 1. 온보딩

온보딩은 사용자가 제품을 처음 만나는 순간입니다. 첫인상을 결정하고, 제품 가치를 전달하며, 사용 방법을 안내합니다.

#### 온보딩의 목표

- 제품의 핵심 가치 전달
- 초기 설정 완료 지원
- 주요 기능 소개
- 사용자 불안감 해소

#### 온보딩 원칙

**1. 가치 우선 (Value First)**
- 기능 나열이 아닌 혜택 강조
```
❌ "실시간 동기화 기능"
✅ "모든 기기에서 최신 파일에 접근하세요"

❌ "고급 분석 도구"
✅ "데이터로 더 나은 결정을 내리세요"
```

**2. 단계별 안내 (Progressive Disclosure)**
- 한 번에 하나씩
- 필요한 시점에 정보 제공
- 압도하지 않기

```
좋은 예:
1단계: "환영합니다! 프로필을 만들어볼까요?"
2단계: "관심 주제를 선택해주세요."
3단계: "준비 완료! 첫 프로젝트를 시작해보세요."
```

**3. 건너뛰기 옵션 (Skip Option)**
- 사용자가 선택할 수 있게
- "나중에", "건너뛰기" 제공
- 강요하지 않기

```
[온보딩 화면]
주요 CTA: "시작하기"
Secondary: "건너뛰기"
```

**4. 진행 상황 표시**
- 몇 단계 중 몇 번째인지
- 얼마나 남았는지
- 사용자가 통제감 느끼게

```
"3단계 중 1단계"
또는 진행률 바: [■■■□□] 60%
```

#### 온보딩 문구 예시

**환영 메시지:**
```
❌ "앱에 오신 것을 환영합니다. 다양한 기능을 제공합니다."
✅ "안녕하세요! 함께 첫 프로젝트를 시작해볼까요?"
```

**권한 요청:**
```
❌ "알림 권한을 허용하세요."
✅ "중요한 업데이트를 놓치지 않도록 알림을 켜주세요."
(왜 필요한지, 어떤 혜택이 있는지 설명)
```

**완료 메시지:**
```
❌ "설정이 완료되었습니다."
✅ "준비 완료! 이제 [핵심 기능]을 사용해보세요."
```

---

### 2. 네비게이션

네비게이션은 사용자가 제품 내에서 길을 찾는 이정표입니다.

#### 네비게이션 라벨 원칙

**1. 명확하고 구체적**
```
❌ "항목"
✅ "내 프로젝트"

❌ "더보기"
✅ "설정"

❌ "기타"
✅ "도움말 및 지원"
```

**2. 사용자 언어 사용**
- 내부 용어 지양
- 사용자가 쓰는 단어 사용

```
내부 용어 ❌        사용자 용어 ✅
CRM                 고객 관리
Dashboard           대시보드 (or 홈)
Repository          저장소 (or 내 파일)
```

**3. 동사 활용 (작업 중심)**
```
작업을 나타낼 때:
- "사진 업로드"
- "친구 초대"
- "리포트 생성"

상태를 나타낼 때:
- "내 프로필"
- "설정"
- "알림"
```

**4. 일관성**
- 같은 의미는 같은 단어
- 스타일 통일 (전부 명사 or 전부 동사)

```
일관성 있음 ✅:
- 홈
- 검색
- 프로필
- 설정

일관성 없음 ❌:
- 홈
- 찾기 (검색과 혼재)
- 내 프로필
- 설정하기
```

#### 브레드크럼 (Breadcrumb)

```
홈 > 전자제품 > 노트북 > 게이밍 노트북

원칙:
- 간결한 레이블
- 계층 구조 명확히
- 클릭 가능 여부 표시
```

### 3. 액션 요소 (Buttons & CTAs)

버튼 텍스트는 사용자가 무엇을 할 수 있는지, 무슨 일이 일어날지 알려줍니다.

#### 버튼 레이블 원칙

**1. 동사로 시작**
- 행동을 명확히

```
❌ "확인"
✅ "삭제하기"

❌ "OK"
✅ "저장"

❌ "제출"
✅ "문의 보내기"
```

**2. 결과 예측 가능**
- 클릭 후 무슨 일이 일어날지

```
❌ "다음"
✅ "결제 단계로"

❌ "보내기"
✅ "이메일 발송"
```

**3. 구체적으로**
```
일반적 ❌          구체적 ✅
"확인"             "회원가입 완료"
"취소"             "변경사항 취소"
"저장"             "프로필 저장"
```

**4. 긍정적 표현 (가능한 경우)**
```
부정적 ❌          긍정적 ✅
"아니요"           "유지하기"
"거부"             "나중에"
```

#### CTA (Call To Action) 작성법

**무료 체험 유도:**
```
❌ "시작"
✅ "무료로 시작하기"
```

**다운로드:**
```
❌ "다운로드"
✅ "무료 앱 다운로드"
```

**구매:**
```
❌ "구매"
✅ "지금 구매하고 10% 할인받기"
```

**가입:**
```
❌ "가입"
✅ "무료로 가입하기" (이미 '무료'임을 강조)
```

#### 파괴적 행동 (Destructive Actions)

삭제, 취소 등 되돌릴 수 없는 행동:

**확인 다이얼로그:**
```
제목: "정말 삭제하시겠습니까?"
설명: "이 작업은 되돌릴 수 없습니다. 프로젝트 'ABC'와 모든 파일이 영구 삭제됩니다."
버튼:
- Primary (위험): "삭제"
- Secondary: "취소"
```

**경고 명확히:**
```
❌ "확인하시겠습니까?"
✅ "삭제하면 복구할 수 없습니다. 계속하시겠습니까?"
```

### 4. 피드백

피드백은 시스템이 사용자 행동에 반응하여 전달하는 메시지입니다.

#### 성공 메시지

**원칙:**
- 무엇이 성공했는지 명확히
- 다음 행동 안내 (선택적)

```
❌ "성공"
✅ "파일이 업로드되었습니다."

❌ "완료되었습니다."
✅ "비밀번호가 변경되었습니다. 새 비밀번호로 로그인해주세요."

❌ "저장됨"
✅ "변경사항이 저장되었습니다."
```

#### 오류 메시지

**좋은 오류 메시지의 3요소:**
1. **무엇이** 잘못됐는지
2. **왜** 발생했는지 (선택적)
3. **어떻게** 해결하는지

```
나쁜 예 ❌:
"오류 발생"
"잘못된 입력"
"실패"

좋은 예 ✅:
"이메일 형식이 올바르지 않습니다. '@'를 포함해주세요."
"비밀번호는 최소 8자 이상이어야 합니다."
"파일 크기가 너무 큽니다. 10MB 이하로 업로드해주세요."
```

**오류 톤:**
- 비난하지 않기
- 공감하기
- 해결 방법 제시

```
비난하는 톤 ❌:
"잘못 입력하셨습니다."
"규칙을 따르지 않았습니다."

공감하는 톤 ✅:
"다시 확인해주세요."
"올바른 형식으로 입력해주세요."
```

**404 오류 (페이지 없음):**
```
❌ "404 Error: Page Not Found"

✅ "페이지를 찾을 수 없습니다."
   "찾으시는 페이지가 삭제되었거나 주소가 변경되었을 수 있습니다."
   [홈으로 돌아가기] [검색하기]
```

**500 오류 (서버 오류):**
```
❌ "Internal Server Error"

✅ "일시적인 오류가 발생했습니다."
   "잠시 후 다시 시도해주세요. 문제가 계속되면 고객센터로 문의해주세요."
   [다시 시도] [고객센터]
```

#### 로딩 및 진행 상황

**로딩:**
```
❌ "로딩 중..."

✅ "파일을 업로드하고 있어요..."
   "데이터를 불러오는 중입니다..."
   "거의 다 됐어요!"
```

**긴 대기 시간:**
```
"시간이 조금 걸릴 수 있어요. 창을 닫지 말고 기다려주세요."
"대용량 파일을 처리하고 있습니다. (약 2분 소요)"
```

### 5. 입력 요소 (Forms)

입력 폼은 사용자가 정보를 제공하는 곳입니다. 명확한 안내가 필수입니다.

#### 레이블 (Labels)

**명확하고 구체적:**
```
❌ "이름"
✅ "이름 (본명)"

❌ "번호"
✅ "휴대폰 번호"

❌ "주소"
✅ "배송 받으실 주소"
```

**필수 표시:**
```
"이메일 *" (필수)
"닉네임 (선택)"
```

#### 플레이스홀더 (Placeholders)

**역할:**
- 입력 예시 제공
- 형식 안내
- 레이블 보완 (대체 아님!)

```
Label: "이메일 주소 *"
Placeholder: "your-email@example.com"

Label: "생년월일"
Placeholder: "YYYY-MM-DD"

Label: "휴대폰 번호"
Placeholder: "010-1234-5678"
```

**주의사항:**
- 플레이스홀더는 사라지므로 중요 정보 배치 금지
- 레이블을 대체할 수 없음

#### 도움말 텍스트 (Helper Text)

```
Label: "비밀번호 *"
Helper: "8자 이상, 영문 대소문자, 숫자, 특수문자 포함"

Label: "초대 코드"
Helper: "초대 코드가 있으면 입력해주세요. (선택사항)"
```

#### 인라인 검증 (Inline Validation)

**실시간 피드백:**
```
[이메일 입력 중]
✅ "사용 가능한 이메일입니다."
❌ "이미 사용 중인 이메일입니다."

[비밀번호 강도]
약함: "비밀번호가 너무 약합니다. 문자와 숫자를 조합해주세요."
보통: "안전한 비밀번호입니다."
강함: "매우 안전한 비밀번호입니다!"
```

### 6. 콘텐츠

콘텐츠 영역은 정보를 전달하는 곳입니다.

#### Empty States (빈 상태)

사용자가 아직 콘텐츠를 생성하지 않았을 때:

**구조:**
1. 상황 설명
2. 이유 또는 혜택
3. 행동 유도

```
예시 1 (처음 사용):
"아직 프로젝트가 없습니다."
"프로젝트를 만들어 팀과 협업을 시작하세요."
[첫 프로젝트 만들기]

예시 2 (검색 결과 없음):
"'키워드'에 대한 검색 결과가 없습니다."
"다른 키워드로 검색하거나 철자를 확인해주세요."
[검색어 지우기]

예시 3 (필터 결과 없음):
"선택한 조건에 맞는 항목이 없습니다."
"필터를 조정하거나 초기화해보세요."
[필터 초기화]
```

#### 설명 텍스트

**스캔 가능하게:**
- 짧은 문단
- 불릿 포인트 활용
- 시각적 계층 구조

```
❌ 긴 문단:
"우리 서비스는 프로젝트 관리를 쉽게 할 수 있도록 도와주는 도구입니다. 팀원들과 실시간으로 협업하고, 작업을 추적하며, 마감일을 관리할 수 있습니다. 또한 다양한 통합 기능을 제공하여..."

✅ 스캔 가능:
**팀 협업을 위한 올인원 도구**

주요 기능:
• 실시간 협업
• 작업 추적 및 관리
• 마감일 알림
• 외부 도구 연동
```

### 7. 커뮤니케이션 (알림, 이메일 등)

#### 푸시 알림

**원칙:**
- 가치 있는 정보만
- 행동 유도
- 개인화

```
❌ "새 업데이트가 있습니다."
✅ "민수님이 댓글을 남겼습니다: '좋은 아이디어네요!'"
```

## 고급 원칙

### 모바일 UX 라이팅 원칙

모바일은 화면이 작고, 사용자가 이동 중일 수 있습니다.

**1. 더 짧게**
```
데스크톱: "변경사항을 저장하시겠습니까?"
모바일: "저장하시겠습니까?"
```

**2. 터치 친화적 레이블**
```
❌ "클릭"
✅ "탭" 또는 "선택"
```

**3. 한 줄에 하나의 아이디어**
```
❌ "파일이 업로드되었으며, 팀원들에게 공유되었고, 알림이 전송되었습니다."

✅ "파일이 업로드되었습니다."
   "팀원들에게 공유되었습니다."
```

### 글로벌 사용자를 위한 현지화

**1. 번역 가능하게 작성**
- 짧은 문장
- 명확한 의미
- 문화적 표현 지양

```
번역 어려움 ❌:
"Let's get the ball rolling!" (관용구)

번역 쉬움 ✅:
"시작해볼까요?"
```

**2. 공간 고려**
- 언어마다 길이 다름
- 독일어는 영어보다 30% 더 길 수 있음
- UI 공간 여유 두기

**3. 문화적 차이**
- 색상 의미 (빨강 = 경고 vs 행운)
- 아이콘 의미 (엄지 up = 좋음 vs 모욕)
- 숫자, 날짜 형식

**4. 변수 사용**
```
하드코딩 ❌:
"You have 3 messages"

변수 사용 ✅:
"{count} messages"
(1 message / 3 messages 자동 처리)
```

### Voice & Tone

**Voice (보이스)**: 브랜드의 일관된 성격
**Tone (톤)**: 상황에 따라 달라지는 감정

#### Voice 정의하기

브랜드를 사람으로 상상:
- 친근한가, 전문적인가?
- 유머러스한가, 진지한가?
- 캐주얼한가, 격식있는가?

```
예시:

Slack의 Voice:
- 친근하고 유머러스
- 캐주얼하지만 프로페셔널
- 긍정적이고 도움이 되는

은행 앱의 Voice:
- 신뢰할 수 있고 안정적
- 명확하고 정확한
- 전문적이지만 따뜻한
```

#### Tone 조절하기

같은 Voice라도 상황에 따라 Tone은 달라집니다.

**성공 시:**
```
Voice 유지, Tone은 축하/긍정적:
"완료되었습니다! 🎉"
```

**오류 시:**
```
Voice 유지, Tone은 공감/해결 중심:
"문제가 발생했네요. 함께 해결해볼까요?"
```

**민감한 상황 (결제 실패, 계정 정지 등):**
```
Voice보다 Tone 우선, 진지하고 명확하게:
"결제가 완료되지 않았습니다. 카드 정보를 확인해주세요."
(유머나 가벼운 표현 지양)
```

#### Voice & Tone 가이드 예시

| 상황 | Voice | Tone | 예시 |
|------|-------|------|------|
| 온보딩 | 친근한 | 환영하는 | "만나서 반가워요! 함께 시작해볼까요?" |
| 성공 | 친근한 | 축하하는 | "완료! 멋지게 해냈어요 🎉" |
| 오류 | 친근한 | 공감하는 | "앗, 문제가 생겼네요. 다시 시도해볼까요?" |
| 경고 | 친근한 | 진지한 | "이 작업은 되돌릴 수 없습니다. 신중히 결정해주세요." |
| 결제 | 친근한 | 신뢰할 수 있는 | "안전하게 결제가 완료되었습니다." |

## UX 라이팅 체크리스트

모든 텍스트를 작성한 후 확인:

- [ ] **명확한가?** 사용자가 즉시 이해하는가?
- [ ] **간결한가?** 불필요한 단어는 없는가?
- [ ] **유용한가?** 사용자가 다음에 무엇을 해야 하는지 아는가?
- [ ] **인간적인가?** 자연스럽고 친근한가?
- [ ] **일관적인가?** 같은 의미는 같은 단어를 사용하는가?
- [ ] **접근 가능한가?** 모든 사용자가 이해할 수 있는가?
- [ ] **브랜드에 맞는가?** Voice & Tone이 일치하는가?
- [ ] **액션을 유도하는가?** 사용자가 무엇을 할 수 있는지 명확한가?
- [ ] **오류를 예방하는가?** 충분한 안내를 제공하는가?
- [ ] **번역 가능한가?** (글로벌 제품인 경우)

---

## 실무 팁

**1. 소리 내어 읽기**
- 자연스럽게 들리는가?
- 실제 대화에서 쓸 표현인가?

**2. 사용자 테스트**
- 사용자가 이해하는가?
- 오해하는 부분은 없는가?

**3. 맥락에서 검토**
- 디자인과 함께 봤을 때 명확한가?
- 흐름이 자연스러운가?

**4. 콘텐츠 우선 디자인**
- 디자인 전에 문구 먼저 작성
- 문구에 맞춰 UI 조정

**5. 스타일 가이드 작성**
- Voice & Tone 정의
- 자주 쓰는 표현 정리
- Do's and Don'ts

**6. 협업**
- 디자이너, 개발자와 긴밀히 협업
- 초기부터 UX Writer 참여




## 과제

### 연구 과제
1. 새로운 앱을 하나 선택해 다운받아 온보딩 문구를 그대로 적어봅니다. 그런 다음 “이 문구가 나의 어떤 불안을 해소하려 했는가?"를 설명해봅니다. 설명이 막히는 지점은 어디인가요?
2. 방금 고른 앱의 메뉴에서 주요 기능의 라벨을 5개를 찾아보고, ‘사용자가 어떤 행동을 기대한다고 가정했는가’를 유추해 봅니다. 설명이 어려운 라벨은 무엇이며, 왜 그렇게 느끼나요?
3. 지금까지 연구했던 제품들 중 하나를 선택해, 핵심 CTA 버튼 1개를 새로 작성해봅니다.  그런 다음 “사용자가 눌렀을 때 어떤 결과를 기대하는가"를 문장으로 설명해봅니다. 설명이 자연스럽게 이어지지 않는 부분은 어디인가요?

### 실전 과제
1. 같은 분야의 기능이 유사한 두가지 경쟁 앱을 선택해 봅니다.
2. 두 서비스의 UX 라이팅을 분석해 해당 서비스가 어떤 입장에서 고객과 소통을 하고 있는지에 대한 캐릭터와 톤을 분석합니다.
3. 전반적인 UX 라이팅 경험을 둘러보고, 7가지 기능 측면에서 잘못된 소통을 하고 있는 부분을 발견해 개선점을 찾아 제안해 봅니다.
